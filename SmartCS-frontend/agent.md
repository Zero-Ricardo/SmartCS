# Agent Working Agreement（必读与记忆设定）

本文件用于约束后续所有 AI / 代理在本项目中的工作方式，核心目标是：每次行动都基于当前文档事实，避免“猜用户怎么在 App 里移动”、“猜技术栈版本”、“猜接口形态”。

## 1. 可读清单（每次任务开始视情况而定）

每次收到新需求、修 bug、补充文档、生成代码或修改任何文件前，视情况而定，可以根据需要读取并以其为准：

1. [PRD.md](file:///d:/Customer%20Service/PRD.md)
   - 定义产品目标、P0/P1功能范围、交互规范、免登录与对话记忆的产品设定。
2. [FLOW.md](file:///d:/Customer%20Service/FLOW.md)
   - 定义用户流程触发条件、逐步序列、决策点、成功/错误行为、屏幕清单与路由规则。
   - 禁止在流程/路由上做“合理猜测”；若文档缺失，优先补文档而不是编造。
3. [TECH_STACK.md](file:///d:/Customer%20Service/TECH_STACK.md)
   - 定义依赖与工具链的“确切版本锁定”、API 契约版本、卡片 schema 与错误码基线。

## 2. 需要长期记忆的设定（不可擅自改变）

以下为当前项目的强约束设定，除非用户明确要求修改，否则必须保持一致：

- **完全免登录可提问**：用户不需要登录即可使用所有对话功能。
- **对话记忆**：重新打开页面/重新打开客服弹窗时，会恢复最近对话（本地存储 + `Visitor ID` 作为会话关键标识）。
- **弹窗形态**：客服以“主界面按钮触发 + 悬浮弹窗对话框”为主形态，不依赖 URL 路由变化（具体见 FLOW 的路由规则）。
- **流式回复**：AI 回复通过 SSE 或 WebSocket 流式接收，并呈现打字机效果。
- **消息渲染（当前仅对话）**：当前版本仅实现对话能力；支持基础富文本消息渲染，结构化消息仅保留（可选）联系方式与满意度；失败必须可降级为纯文本。
- **技术栈定向**：Vue 3 + Element Plus + Pinia；依赖版本以 TECH_STACK 为唯一基准。

## 3. 行为准则（每次动作如何做）

### 3.1 先文档，后代码
- 任何实现或改动必须能在 PRD/FLOW/TECH_STACK 中找到依据。
- 若用户需求与文档冲突：优先更新文档（说明变更点），再做实现。


### 3.2 不猜测用户移动路径
- 所有“用户如何到达某处/按了什么按钮/会跳什么路由”的描述必须引用 FLOW 的屏幕清单与流程索引。
- 需要新增/调整流程时：先改 FLOW，再改代码或 PRD。

### 3.3 版本必须可复现
- 新增任何依赖、API、工具链配置前必须检查 TECH_STACK：
  - 若已存在：严格按锁定版本使用。
  - 若不存在：先把版本与用途写入 TECH_STACK，再引入依赖或写实现方案。

### 3.4 错误处理必须“可见且可恢复”
- 任何网络/流式/渲染/上传/转人工失败，必须在 FLOW 对应流程的“错误时发生什么”中有明确行为。
- 没有定义的错误分支：先补 FLOW，再做实现。

### 3.5 工程分层与解耦规范（防巨石代码）
- 目录分层建议：`src/app`（应用装配）/ `src/pages`（页面）/ `src/widgets`（复合业务块）/ `src/features`（单一业务能力）/ `src/entities`（领域模型）/ `src/shared`（通用能力）。
- 单文件体量约束：组件建议 ≤ 300 行；超过阈值必须拆分为容器组件 + 展示组件 + hooks/composables。
- 状态边界：Pinia 仅存放跨页面或跨模块共享状态；组件私有状态留在组件内；禁止把临时 UI 状态塞进全局 store。
- 依赖方向：`shared -> entities -> features -> widgets -> pages -> app` 单向依赖；下层不得反向依赖上层。
- 业务与视图分离：网络请求、协议适配、数据转换放 `services`/`adapters`；Vue 组件只做渲染与交互编排。
- 接口与类型治理：所有 API 请求/响应定义在 `src/shared/api` 与 `src/shared/types`；卡片 schema 校验统一在适配层处理。
- 命名规范：组件 `PascalCase`，组合式函数 `useXxx`，store `useXxxStore`，常量 `UPPER_SNAKE_CASE`，文件名与导出名一致。
- 评审最小检查项：是否符合分层、是否出现超大组件、是否有重复逻辑可下沉、是否补齐错误分支、是否更新 FLOW/TECH_STACK/progress。

## 4. 文档变更规则（谁是事实源）

当多份文档信息不一致时，按以下优先级裁决并修复文档一致性：

1. FLOW（流程与路由事实源，防猜测）
2. TECH_STACK（版本与契约事实源）
3. PRD（需求范围与优先级事实源）

修复方式：
- 更新冲突的文档段落，确保三者互相引用时不矛盾。
- 变更必须写清楚“旧设定 → 新设定”，并说明影响的 Flow ID 或依赖版本。

## 5. 输出要求（交付时必须包含）

每次交付（无论是文档还是代码）必须提供：
- 本次读取的文档列表（只列文件名即可）
- 本次变更的文件清单
- 若涉及流程：指出影响的 Flow ID
- 若涉及依赖/接口：指出影响的 TECH_STACK 条目（版本/契约）
- 不要保存测试用例或临时代码
